# 用户角色研究与描述

## 用户角色概览

本平台主要服务三类用户：学生、教师和管理员。下面详细介绍每个用户角色。

---

## 1. 学生用户角色

### 角色名称: 小明（学生）

#### 基本信息
- **姓名**: 小明
- **年龄**: 20岁
- **身份**: 计算机科学与技术专业 大二学生
- **学号**: 2024001234
- **专业**: 计算机科学与技术
- **年级**: 2024级
- **班级**: 01班

#### 性格特征
- 年轻、活力、好学
- 技术接受度高
- 重视学习和生活效率
- 喜欢参与校园活动

#### 核心需求
1. **学业需求**
   - 快速查看课表
   - 方便地选课
   - 及时了解成绩
   - 获取课程相关信息

2. **生活需求**
   - 借阅图书方便
   - 了解校园动态
   - 参与社区互动
   - 获取有用信息

3. **效率需求**
   - 操作步骤少
   - 加载速度快
   - 界面易于使用
   - 快速找到信息

#### 使用场景

**早上8点 - 查看课表**
```
场景: 小明早上起床后，需要确认今天的课程
行为: 打开平台 → 点击课程管理 → 查看课表 → 确认今天没有上午课
痛点: 之前需要打开学校官网，非常麻烦
```

**下午2点 - 选课期间**
```
场景: 选课期间开放，小明需要进行选课操作
行为: 进入选课界面 → 搜索感兴趣的课程 → 查看详情 → 加入购物车 → 确认选课
痛点: 同时选择多门课程时容易出错
```

**晚上7点 - 查询成绩**
```
场景: 教师已发布考试成绩
行为: 进入成绩查询 → 查看各科成绩 → 查看详细信息 → 分享到社区
痛点: 想了解自己的排名和GPA
```

**晚上9点 - 校园社区互动**
```
场景: 刷校园社区
行为: 查看热门话题 → 阅读帖子 → 点赞评论 → 分享心得
痛点: 希望有一个地方分享学习心得和校园趣事
```

- 重视教学质量

2. **互动需求**

   - 获得学生反馈
   - 发现优秀学生

3. **效率需求**

**上班10点 - 发布课程资源**
```
场景: 今天要给学生发布新的课程资料

**中午12点 - 评定成绩**
```
场景: 需要批量评定学生的作业成绩
行为: 进入成绩评定 → 查看学生列表 → 输入成绩 → 批量保存 → 发送反馈
痛点: 成绩众多，手工输入费时

行为: 查看通知 → 进入讨论 → 回复学生问题 → 标记已解答
痛点: 分散的问题难以统一管理
```

```

## 用户对比分析

| 维度 | 学生用户 | 教师用户 | 管理员用户 |
|------|---------|---------|----------|
| 使用频率 | 高 (日常) | 中 (工作时) | 中 (定期) |
| 技术水平 | 高 | 中 | 中 |
| 关键功能 | 课表、选课、成绩 | 课程管理、成绩评定 | 通知、审核、数据 |
| 核心痛点 | 效率低、步骤多 | 工作量大、管理难 | 权限管理、数据一致 |
| 设计重点 | 简洁、快速 | 完善、高效 | 安全、可靠 |

---

## 总结

通过用户角色研究，我们了解到不同用户有不同的需求和使用场景。设计时需要：

1. **学生用户**: 重视界面简洁、操作快捷、功能实用
2. **教师用户**: 重视功能完善、管理高效、反馈及时
3. **管理员用户**: 重视权限安全、数据准确、控制完善

原型设计应充分考虑这三类用户的需求，提供相应的功能模块和交互设计。
